## Accessibility

This text input component is built on the USWDS text input and satisfies the following WCAG 2.1 AA criteria:

- **1.3.1 Info and Relationships** — Uses a native `<input>` element associated with a `<label>` via matching `id` and `htmlFor` attributes
- **1.4.3 Contrast (Minimum)** — Meets a 4.5:1 contrast ratio between text and background in all states including error and success
- **2.1.1 Keyboard** — Fully operable via keyboard — focusable and navigable without a mouse
- **2.4.7 Focus Visible** — Displays a visible focus indicator when the input receives keyboard focus
- **3.3.1 Error Identification** — When in an error state, `aria-invalid="true"` is applied and error messaging is connected via `aria-describedby` — error is never communicated by color alone
- **3.3.2 Labels or Instructions** — Every input has a persistent visible label that does not disappear on focus
- **4.1.2 Name, Role, Value** — Uses a native `<input>` element so name, role, and state are communicated correctly to screen readers

## Usage Guidelines

- Always pair a TextInput with a Label — never use placeholder text as a substitute for a label
- Use the correct `type` attribute for the expected input — `email` for email addresses, `tel` for phone numbers, `number` for numeric input — this triggers the correct keyboard on mobile and enables browser validation
- Use `validationStatus="error"` alongside a visible error message component — the error state alone is not sufficient to communicate what went wrong
- Use `validationStatus="success"` sparingly — reserve it for cases where confirming valid input is genuinely useful to the user
- Use `readOnly` for values the user should see but not edit — for example a pre-filled field populated from another source
- Do not use `disabled` when `readOnly` is more appropriate — disabled fields are not announced by screen readers and cannot receive focus
- Keep input width proportional to the expected length of input — a zip code field should be narrower than a street address field
