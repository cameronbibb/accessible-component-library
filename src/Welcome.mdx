# Kentucky Personnel Cabinet — Accessible Component Library

This component library and design system was developed in response to the Kentucky Personnel Cabinet's Division of Technology Services initiative to modernize legacy web applications and establish a unified, accessible UI standard across all internal tools and public-facing applications.

It is built on the [U.S. Web Design System (USWDS)](https://designsystem.digital.gov/) and implements [WCAG 2.1 Level AA](https://www.w3.org/WAI/standards-guidelines/wcag/) accessibility standards, in compliance with the ADA Title II rule effective April 2026 for state and local governments.

---

## Purpose

The Personnel Cabinet currently maintains a portfolio of web applications built independently over time, resulting in visual inconsistency, varied user experiences, and accessibility gaps. This design system addresses those problems by establishing:

- A **single source of truth** for UI components, design tokens, and usage guidelines
- **WCAG 2.1 AA compliance** built into every component from the ground up
- A **consistent visual language** across all applications — colors, typography, spacing, and interactive patterns
- A **reusable component library** that reduces development time and ensures new applications inherit accessibility and consistency by default

## How to Use This Library

### Installation

Clone the repository and install dependencies:

```bash
git clone https://github.com/cameronbibb/accessible-component-library.git
cd accessible-component-library
npm install
```

### Running Storybook

```bash
npm run storybook
```

### Importing Components

Import components directly from the `src/components` directory:

```tsx
import { Button } from "./components/Button/Button";
import { TextInput } from "./components/TextInput/TextInput";
```

## Design Principles

**Accessibility first.** Every component is built to WCAG 2.1 AA standards. Accessibility is not a retrofit — it is a requirement built into the foundation.

**Consistency over customization.** Components define the boundaries of the UI. New applications draw from this library rather than making independent design decisions.

**Stability over novelty.** This system is built for long-term maintainability in a government context. Dependencies are chosen for reliability, not trendiness.

**Documentation as a deliverable.** Every component includes usage guidelines, accessibility criteria, and implementation notes. The documentation is as important as the code.

## Development Standards

- Components are wrapped in a facade pattern — consuming applications import from this library, not directly from USWDS. This isolates dependency changes to a single location.
- All components are tested against WCAG 2.1 AA criteria using the Storybook accessibility addon.
- Color contrast ratios meet or exceed the 4.5:1 minimum for normal text required by WCAG 1.4.3.
- All interactive elements have visible focus indicators per WCAG 2.4.7.
- Form inputs use proper label association, ARIA attributes, and error messaging patterns per WCAG 1.3.1 and 3.3.1.
